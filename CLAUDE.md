## 構成

- `index.html` … サイト本体（HTML + ページ固有のインラインCSS/JS、外部フレームワーク不使用）
- `theme.css` / `theme.js` … 全ページ共通の基盤（カラーパレット・ダークモード・リセット・
  テーマ切替）。**色やテーマ挙動の変更はここだけを編集**し、各ページには書かない。
  theme.js は描画前に実行する必要があるため `<head>` で同期読み込み（defer/async 禁止）。
- `llms.txt` … AIO用の会社サマリ（内容変更時は index.html と同期）
- `robots.txt` / `sitemap.xml` … SEO。URLを増やしたら sitemap を更新
- `CNAME` … `yaenowa.co.jp`（apex）
- `LOGO.png` / `README.md`

## 必須ルール（契約・法務）— コンテンツ編集時に厳守

1. **実績・事例の書き方** 🔴
   - 受託業務は秘密保持義務の対象。**特定案件を描写しない**。
   - 出さない要素: 客先名、**業界名**、規模を示す語（「大手」等）、「複数ブランド」など案件を特定し得る表現。
   - 語り口は「過去に○○を担当した」ではなく、**「○○に対応できる（提供領域・スキル）」**に保つ。
   - 例外: 当該パートナー企業から**書面による事前同意**を得た範囲のみ、具体的な記述を許容。
2. **成果物を載せない** 🔴
   - 要件定義書・設計書・図・画面・コード等の**実物・抜粋・スクリーンショットは掲載不可**
     （著作権は委託元に帰属するため）。
3. **資格・認定** 🟡
   - 取得（合格）が確認できたものだけ掲載する。取得前・受験中の資格は載せない。
   - 例: 生成AIパスポートは合格後に、保有資格として表示＋構造化データ（hasCredential）化。
4. **機密の取り扱い** 🔴
   - 契約書・NDA本文、条項、客先名などをリポジトリにコミット／サイトに掲載しない。

> 上記は実務・構造面のリスク低減を目的とした運用ルール。**法的拘束力のある最終判断は弁護士に確認**すること。

## セキュリティ規約

- 静的サイト（バックエンド・フォーム・認証なし）。攻撃面は小さい。
- 外部リンク（`target="_blank"`）には必ず `rel="noopener"` を付ける。
- 新しい外部リソース（スクリプト・フォント等）を追加する際は提供元の信頼性を確認する。
- インラインJSにユーザー入力を流し込まない（現状XSSベクタなし）。

## SEO / AIO 運用メモ

- 正規URLは **apex ドメイン**（`https://yaenowa.co.jp/`、www無し）。canonical・OGP・sitemap・構造化データを統一すること。
- コンテンツを変更したら **`llms.txt` と JSON-LD 構造化データを同期**する。
- セクション/ページを増やしたら `sitemap.xml` と必要に応じてナビ・FAQ構造化データを更新。
- 構造化データの可視性: FAQ等は**表示されている内容と構造化データを一致**させる（Googleポリシー）。
- アクセシビリティ: 装飾要素は `aria-hidden`、`prefers-reduced-motion` を尊重。

## 開発フロー

- 作業ブランチで開発 → コミット → push。`main` へは PR 作成後にマージ。
- **push が完了したら、ユーザーの指示を待たずに PR を作成してそのままマージしてよい**（自動マージ運用）。
  ただしコンテンツ（実績・資格・会社情報など必須ルールに関わる文言）を変更した場合は、
  下記サブエージェントのレビューを済ませ、🔴 指摘が残っていない状態でマージすること。
  マージ後は PR の URL を報告する。
- コンテンツ変更時は実機（ブラウザ）で表示確認し、JSON-LD の構文を検証する。
- コンテンツ変更後は必ずサブエージェントでレビューする（`.claude/agents/`）:
  - `content-legal-review` … 必須ルール（契約・法務）への違反チェック
  - `site-consistency-check` … JSON-LD 構文・4箇所同期・URL 正規化・sitemap・noopener の検証
